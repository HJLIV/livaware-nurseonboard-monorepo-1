# Continuous Learning

Capture insights from completed work to improve future performance.

## When to Use

- After completing a project or feature
- When encountering a recurring problem
- Building runbooks for common operations
- Conducting retrospectives

## Learning Capture Process

### 1. Identify the Learning
- What worked well?
- What didn't work as expected?
- What would you do differently?
- What surprised you?

### 2. Categorize
- **Technical**: Code patterns, tools, configurations
- **Process**: Workflow improvements, communication
- **Domain**: Business logic insights, user behavior
- **Debugging**: Common issues and their solutions

### 3. Document
Record in a structured format:

```markdown
## Learning: [Title]

**Date**: [When]
**Context**: [What project/task]
**Category**: [Technical/Process/Domain/Debugging]

### What Happened
[Brief description of the situation]

### Key Insight
[The main takeaway]

### Action Items
- [ ] [Concrete change to make]
- [ ] [Process to update]
- [ ] [Knowledge to share]
```

### 4. Apply
- Update relevant skills or documentation
- Share with team members
- Create or update runbooks
- Improve automation or tooling

## Runbook Template

```markdown
# Runbook: [Operation Name]

## Purpose
[What this runbook is for]

## When to Use
[Triggers or situations]

## Prerequisites
- [Required access]
- [Required tools]

## Steps
1. [Step with expected output]
2. [Step with expected output]

## Troubleshooting
### [Symptom]
- Likely cause: [X]
- Fix: [Y]

## Rollback
[How to undo if something goes wrong]
```

## Anti-Patterns
- "We'll remember this" — always write it down
- Documenting without actionable takeaways
- Learning capture without follow-through
- Blaming individuals instead of identifying systemic issues
